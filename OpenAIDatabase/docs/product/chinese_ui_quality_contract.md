# Memory Atlas 中文 UI 质量合同

更新时间：2026-07-01

适用版本：Memory Atlas v1.1.6 Stage 0 Phase 0.1

本合同定义 Memory Atlas 中文界面的编码、文案、布局承载和可读性底线。它是后续首页、Inspector、搜索、复盘、数据导图、记忆时间河和记忆星系改造的前置合同；没有通过本合同的页面不能进入视觉增强或发布验收。

## 1. 目标

Memory Atlas 必须先做到“能看懂、能追溯、能判断下一步”。中文 UI 质量不是装饰项，而是系统可信度的基础门槛。

本阶段要求：

1. 所有新增 Markdown、JSON、UI 文案和导出说明使用 UTF-8。
2. 页面不得出现替换字符、mojibake、未知编码残留、英文占位、不可读标签。
3. 中文标题、按钮、卡片、Inspector 标签和状态文案保持一致。
4. 长摘要、证据说明、公式解释进入展开层或详情层，不塞入表格和按钮。
5. 低宽度视口下不得出现关键中文文案重叠、截断到不可理解或横向溢出。

## 2. 适用范围

必须覆盖：

- 默认首页 / 记忆总览。
- Galaxy / 记忆星系。
- Timeline / 记忆时间河。
- Data Map / 数据导图。
- ROI Dashboard。
- Search / Review / Summary / Iteration。
- Inspector。
- Proposal-only 调整区。
- Markdown 报告、导出说明、截图验收文档。

不在本阶段直接修改：

- 核心前端实现。
- CSS 布局修复。
- 运行时数据生成。
- raw/private 数据读取。
- 长期记忆写回。

上述实现项只在本合同中定义验收标准，后续 phase 按合同实施。

## 3. 编码与文本来源规则

### 3.1 文件编码

- 新增文档必须为 UTF-8。
- 禁止提交带有典型 mojibake 的文本，例如 replacement character `U+FFFD`、Latin-1 残留 `U+00C2/U+00C3`、或 UTF-8 被错误解码后形成的三字符乱码片段。
- RTF、HTML、CSV 或 JSON 导出进入仓库前必须转换为 UTF-8 文本或明确作为外部输入，不得直接作为 UI 文案源。
- 所有中文 Markdown 标题必须能被普通文本查看器、GitHub 和本地预览一致显示。

### 3.2 UI 文案来源

UI 文案必须来自下列之一：

1. 明确的 React/TypeScript 文案常量。
2. redacted derived snapshot 中的人类可读字段。
3. 产品合同或验收文档中的固定标签。

禁止：

- 从 raw/private transcript 直接取文案。
- 把内部字段名直接暴露给用户，例如无解释的 `cluster_id`、`source_ref`、`debug_payload`。
- 在用户主界面显示未解释的英文占位或空值占位，除非处于 Debug / Agent Inspector 且带中文解释。

## 4. 中文标签规范

### 4.1 导航标签

| 英文概念 | 中文主标签 | 允许辅助说明 |
|---|---|---|
| Overview | 记忆总览 | 今日状态、建议动作、机会和风险 |
| Galaxy | 记忆星系 | 主题引力源、星云、轨迹 |
| Timeline | 记忆时间河 | 主题演化、事件脉冲、决策节点 |
| Data Map | 数据导图 | 来源、主题、资产、行动 |
| Search | 搜索 | 匹配原因、证据、跳转 |
| Review | 复盘 | 本期变化、衰退、机会 |
| Summary & Iteration | 总结与迭代 | 下一步行动、可导出草案 |
| Inspector | 详情解释 | 证据、公式、关联、proposal |
| Proposal | 调整草案 | 不直接写入长期记忆 |

### 4.2 状态标签

| 状态 | 中文显示 | 说明 |
|---|---|---|
| current | 当前有效 | 仍可使用 |
| needs_review | 需要复核 | 时间或证据不足 |
| stale_short_term | 短期过期 | 临时信息已过窗口 |
| black_hole | 低价值循环 | 高投入低收益或反复返工 |
| proto_star | 新生机会 | 早期但可能高 ROI |
| proposal_only | 仅生成草案 | 不写长期记忆 |
| redacted | 已脱敏 | 隐私内容不可见 |

## 5. 文案长度和布局承载规则

### 5.1 卡片

- 卡片标题建议不超过 18 个中文字符。
- 卡片摘要建议不超过 2 行。
- 超过 2 行的解释进入展开层、Inspector 或详情页。
- 卡片必须保留一个可读的主结论，不得只显示指标。

### 5.2 表格

表格只承载：

- 短标签。
- 数字。
- 状态。
- 日期。
- 低长度摘要。

表格禁止承载：

- 长段落。
- 公式解释。
- 证据全文。
- 多句行动建议。
- 多层级嵌套原因。

长内容必须进入：

- Inspector。
- expandable details。
- hover card。
- side panel。
- Markdown report。

### 5.3 按钮和控件

- 按钮文案必须表达动作，例如 `查看证据`、`生成草案`、`打开时间河`。
- 禁止把说明句塞进按钮。
- 模式切换使用稳定标签：`展示模式`、`分析模式`。
- proposal 相关按钮必须明确“不直接写入长期记忆”。

### 5.4 Inspector

Inspector 必须优先显示人类可读解释：

1. 这是什么。
2. 为什么重要。
3. 证据来自哪里。
4. 关联到哪些主题、资产、行动。
5. 公式或参数如何影响结果。
6. 可以生成什么 proposal。

Debug / Agent 字段必须默认折叠。

## 6. 低宽度视口规则

移动或窄宽度视口下必须满足：

- 主要导航和页面标题不重叠。
- 卡片标题不断到无法理解。
- 表格可改为列表、分组或横向受控滚动。
- 图表控件不遮挡主画布。
- Inspector 可关闭、可返回主视图。
- 长英文 token、路径、ID 使用换行或截断中间显示，不能撑破容器。

## 7. 验收门槛

Phase 0.1 只定义合同，验收方式如下：

1. 文档存在且为 UTF-8。
2. 文档不包含典型乱码字符。
3. 合同覆盖编码、中文标签、文案长度、表格长句限制、Inspector、proposal-only 提示和低宽度视口。
4. 后续实现 phase 必须用 Playwright 或等效浏览器截图证明中文显示可读。
5. 若问题来自 CSS、字体或浏览器渲染，必须登记为后续实现任务，不能把责任归为用户不会操作。

## 8. 回滚

本合同不改变运行时行为。若合同内容错误，回滚方式为：

1. 删除或恢复本文件。
2. 删除对应验收文件中引用的 Phase 0.1 条目。
3. 在开发记录中标记合同撤回原因。
