# Memory Atlas 页面视觉密度基线

更新时间：2026-07-01

适用版本：Memory Atlas v1.1.6 Stage 0 Phase 0.2

本文件定义 Memory Atlas 各核心页面的视觉密度基线。它用于防止后续实现把系统退化为列表、表格、静态卡片或普通 dashboard。Phase 0.2 只建立验收合同，不替换现有 UI、不启动浏览器截图、不读取 raw/private 数据。

## 1. 目标

Memory Atlas 的核心体验必须是可视化记忆系统，而不是文档索引或指标列表。每个板块都需要一个默认视觉主区，列表只能作为 drill-down、证据摘要或 Inspector 内容。

本阶段冻结四个基础视觉化门槛：

| 板块 | 最低视觉化程度 | 主视觉必须回答的问题 |
|---|---:|---|
| 记忆总览 | >= 70% | 当前记忆状态是什么，下一步该做什么 |
| 记忆星系 | >= 90% | 主题、机会、风险和层级如何形成空间关系 |
| 记忆时间河 | >= 85% | 主题、事件、决策和机会如何随时间变化 |
| 数据导图 | >= 80% | 数据如何从来源变成主题、资产和行动 |

视觉化程度是人工验收指标，不是伪精确的运行时公式。它用于阻断明显退化，而不是替代真实截图、交互和用户判断。

## 2. 视觉密度定义

视觉密度指首屏和默认页面中“能直接帮助用户理解状态、结构、时间或行动关系”的视觉主区占比。

计入视觉密度：

- 星系、星云、流场、轨迹、地形、引力源。
- 时间河、主题带、事件脉冲、决策节点、black hole band、proto-star marker。
- 数据导图节点、层级、边、分组、流向。
- 记忆总览中的 Memory Weather、Mini Starfield、River Pulse、状态分布和行动入口。
- 可解释的颜色、大小、位置、轨迹、密度、层级。

不计入视觉密度：

- 普通卡片堆叠。
- 表格。
- 长文本说明。
- 只有数字的 KPI。
- 只有点和边线、没有语义视觉层的普通图。
- 装饰性背景，不映射数据或状态。

## 3. 页面基线

### 3.1 记忆总览

最低门槛：`>= 70%`。

必须包含至少三个视觉主区：

1. Memory Weather 主状态。
2. Mini Starfield 或等效主题空间预览。
3. River Pulse 或等效时间变化预览。

允许辅助：

- 建议动作卡片。
- 层级资产摘要。
- 主题分类摘要。
- 低价值循环和新生机会状态条。

失败条件：

- 首屏只有欢迎语、列表、KPI 卡片或导航说明。
- 建议动作没有可见优先级、原因或跳转。
- Mini Starfield 和 River Pulse 仅是静态占位。

### 3.2 记忆星系

最低门槛：`>= 90%`。

必须包含：

1. 黑底深空主画布。
2. 星云或主题密度层。
3. Flow Field 或轨迹层。
4. cluster 引力源。
5. black hole 风险信号。
6. proto-star 机会信号。
7. Memory Terrain 或等效语义地形层。

失败条件：

- 默认视图像普通 Obsidian Graph。
- 只有点、边线或随机粒子。
- 没有星云、流场、轨迹、黑洞或新生机会。
- Analysis Mode 不能解释视觉信号来源。

### 3.3 记忆时间河

最低门槛：`>= 85%`。

必须包含：

1. 时间河主流。
2. 主题带或 lane。
3. 事件脉冲。
4. 决策节点。
5. black hole band。
6. proto-star marker。
7. zoom / brush / hover / click Inspector 的交互入口。

失败条件：

- 默认退回日期列表、表格或静态散点。
- 不能显示主题随时间增强、衰退或分叉。
- 不能通过 hover/click 看证据摘要。
- reduced motion 后功能不可用。

### 3.4 数据导图

最低门槛：`>= 80%`。

必须包含四层结构：

1. 来源层。
2. 主题层。
3. 资产层。
4. 行动层。

每张导图卡片必须至少能表达：

- label。
- type。
- strength。
- trend。
- evidence_count。
- action_count。
- inspector_link。

失败条件：

- 只是卡片列表或数据源列表。
- 看不出来源如何变成主题、资产和行动。
- 没有 Inspector 跳转。
- 没有折叠或层级控制，导致信息密度失控。

## 4. 截图验收矩阵

后续实现 phase 必须补截图证据。本 phase 只冻结矩阵。

| 页面 | Desktop 1440x900 | Tablet 768x1024 | Mobile 390x844 | 必须截图内容 |
|---|---|---|---|---|
| 记忆总览 | required | required | required | Weather、Mini Starfield、River Pulse、建议动作 |
| 记忆星系 | required | required | required | 星云、流场、黑洞、新生机会、地形层 |
| 记忆时间河 | required | required | required | 时间河、主题带、事件脉冲、brush/hover 入口 |
| 数据导图 | required | required | required | 四层结构、卡片字段、Inspector link |

截图必须保存或引用到后续验收记录，不能只说“人工看过”。

## 5. 反退化规则

以下任一情况直接判失败：

1. 视觉主区空白。
2. 视觉主区只剩列表、表格或普通卡片。
3. Canvas/SVG 存在但没有可见数据承载。
4. 视觉信号无法解释来源、主题、资产、行动或时间意义。
5. 移动端主控件遮挡主画布。
6. 文本重叠导致主结论不可读。
7. 默认页面只能通过 Debug 字段理解。

## 6. Phase 0.2 通过条件

本 phase 通过需要同时满足：

1. 本文件存在且为 UTF-8。
2. 明确四个核心板块的视觉化程度门槛。
3. 明确每个板块的必备视觉主区和失败条件。
4. 明确后续截图矩阵。
5. 明确反退化规则。
6. 开发记录登记本 phase 的目标、范围、风险、验收和回滚。
7. 不新增核心 UI 实现，不读取 raw/private，不直接写长期记忆。

## 7. 回滚

本文件不改变运行时行为。若合同内容错误，回滚方式为：

1. 删除 `docs/acceptance/visual_density_baseline.md`。
2. 移除交付记录、模型参数和开发记录中的 Phase 0.2 条目。
3. Stage 0 状态回到 Phase 0.1 完成、Phase 0.2 未完成。
